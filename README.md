# Credit Card Transaction Clustering

## Research Question

**Do credit-card transactions form natural and density-distinct subpopulations, and do small clusters or anomalous observations meaningfully overlap fraudulent transactions?**

This project performs a complete end-to-end clustering analysis on the UCI Credit Card Fraud Detection dataset (DS-10) to answer this question. The analysis is split into three research phases, followed by a delivery layer that makes the results reproducible, monitorable, and interactive.

---

# Project Phases

## Phase 1 – Data Preparation & Clustering Tendency

- **Ingestion:** Reliable download from the UCI repository with chunked reading.
- **Cleaning:** Removal of exact duplicates, schema validation, and handling of malformed records.
- **Feature Engineering:**
  - All PCA-transformed features (`V1`–`V28`) are retained.
  - `Amount` is log-transformed using `log(1 + Amount)` to reduce skewness.
  - `Time` is encoded using cyclic (`sin`/`cos`) features.
- **Scaling:** StandardScaler and RobustScaler are compared.
- **Dimensionality Reduction:** PCA (≥95% explained variance) and UMAP for visualization.
- **Clustering Tendency:** Hopkins Statistic and VAT analysis.

---

## Phase 2 – Core Clustering Portfolio

Four clustering families are implemented and compared.

| Family | Algorithm |
|---------|-----------|
| Partitioning | K-Means |
| Hierarchical | Agglomerative Clustering |
| Density-Based | DBSCAN / HDBSCAN |
| Model-Based | Gaussian Mixture Models |

- Elbow Method
- Average Silhouette
- Gap Statistic
- Internal evaluation

  - Davies–Bouldin Index
  - Calinski–Harabasz Index
- External evaluation
  - ARI
  - NMI
  - AMI
  - Purity
- Stability analysis
- Bootstrap robustness
- Algorithm agreement heatmaps

## Phase 3 – Advanced Ensemble & Consensus Clustering

The selected advanced technique is **Ensemble and Consensus Clustering (Track 4).**

### Base ensemble generation

Base clusterings are generated by varying:

- clustering algorithm
- number of clusters
- PCA dimensionality
- random initialization
- covariance type (for GMM)

### Consensus construction

A co-association matrix is computed, where each element records how frequently two observations appear in the same cluster across all base models.

Average-linkage hierarchical clustering is then applied on

```
1 − CoAssociation
```

to obtain the final consensus clustering.

### Cluster interpretation

The pipeline automatically generates

- Cluster profiles
- Feature z-score summaries
- Cluster labels
- Exemplars
- Boundary observations
- Decision-tree rules
- Optional SHAP explanations
- Cluster anomaly ranking
- Preprocessing sensitivity analysis

---

## Important Limitation

Consensus clustering does **not** naturally support prediction for unseen observations.

Therefore, new transactions are assigned using the **nearest consensus-cluster centroid in PCA space**.

This approximation is intended for deployment and monitoring and is **not an exact reconstruction** of the original consensus clustering algorithm.

# Reproducibility

The project is fully reproducible.

- Random seeds are stored in `params_phase*.yaml`.
- Artifacts are protected using SHA-256 hashes.
- Fraud labels (`Class`) are **never used** during
  - preprocessing
  - feature engineering
  - clustering
  - hyperparameter tuning
  - consensus construction
  - anomaly detection
  - deployment assignment

The labels are used **only** for post-hoc external evaluation.
